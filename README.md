## What is this?Вє

A proof of concept that demonstrates how a **knowledge graph** architecture can improve predictive models for emergency services by making relationships between entities directly queryable as features.

Inspired by Steve Hedden's [Film Club Knowledge Graph](https://github.com/SteveHedden/fckg) вЂ” an architecture for predicting the Oscars using a 4-layer semantic data stack. This project applies the same pattern to San Francisco's multi-agency emergency response system.

### The 4-layer architecture

```
Layer 4 В· Predictive Models в”Ђв”Ђв”Ђв”Ђ Severity | Multi-agency | Resolution time
Layer 3 В· Enrichment в”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђ Temporal features (hour, day, season, weekend, rush hour)
Layer 2 В· Inference в”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђ isMultiAgency | locationHistory | severityLevel | responseTimeStats
Layer 1 В· Foundation в”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђ OWL ontology (15 classes) + SF Open Data (7.2M dispatch records)
```

### Key result

The knowledge graph's value depends on the prediction task:

| Task | Flat table | Knowledge graph | Delta |
|---|---|---|---|
| Multi-agency escalation (accuracy) | 83.9% | **100.0%** | **+19.2%** |
| Resolution time (MAE) | 24.75 min | **23.94 min** | **+3.3%** |
| Severity classification (accuracy) | **86.9%** | 85.1% | в€’2.2% |

**Why?** Multi-agency status requires traversing three levels of the graph (Incident в†’ Response в†’ Unit в†’ Agency). That relationship doesn't exist in a flat CSV вЂ” it has to be derived from the graph structure. For severity, the signal is already in raw fields (priority, response count), so the graph adds dimensionality without proportional information gain.

## Quick start

### Prerequisites

- Python 3.8+
- No external packages required вЂ” all scripts use standard library only

### 1. Download the data

Download the **Fire Department Calls for Service** CSV from SF Open Data:

https://data.sfgov.org/d/nuek-vuh3 в†’ Export в†’ CSV

Save as `data/raw/fire_calls_for_service.csv`

### 2. Run the pipeline

```bash
# Layer 1: Convert CSV to RDF (generates ~118M triples)
python3 scripts/csv_to_rdf.py \
  --input data/raw/fire_calls_for_service.csv \
  --output data/rdf/

# Layer 2: Derive inferred facts (~23M additional triples)
python3 scripts/inference.py \
  --input data/rdf/ \
  --output data/rdf/inferred/

# Layers 3+4: Temporal enrichment + train models + compare KG vs flat
python3 scripts/predict.py \
  --rdf-dir data/rdf/ \
  --inferred-dir data/rdf/inferred/ \
  --limit 20000

# (Optional) Export data for the interactive map
python3 scripts/export_map_data.py \
  --rdf-dir data/rdf/ \
  --inferred-dir data/rdf/inferred/ \
  --output docs/map_data.json \
  --limit 200000
```

Use `--limit` to test with a subset first. The full dataset (7.2M rows) takes several minutes for Layer 1 and requires significant RAM for Layer 2.

### 3. View the interactive visualizations

```bash
cd docs/
python3 -m http.server 8080
```

Open http://localhost:8080 for the ontology viewer, or http://localhost:8080/map.html for the incident map.

## Ontology overview

The central entity is `Incident`, identified by its CAD number вЂ” the universal join key across all agency datasets. The `IncidentConcept` class enables cross-agency alignment: Fire's "Traffic Collision" and Police's "Vehicle Accident" map to the same abstract concept.

